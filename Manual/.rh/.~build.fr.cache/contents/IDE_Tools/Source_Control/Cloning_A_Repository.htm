La raison en est que lorsque vous livrez, vous stockez vos modifications localement, ce qui vous permet de travailler loin d'une connexion Internet et de vous synchroniser lorsque vous êtes prêt. et de vous synchroniser quand vous êtes prêt, et pour vous synchroniser, vous poussez vos changements vers le dépôt distant. L'extraction des modifications signifie que vous voulez extraire toutes les modifications que d'autres personnes ont apportées depuis le référentiel distant vers votre référentiel local. Cependant, cela signifie qu'il peut y avoir des moments où des personnes peuvent éditer les mêmes fichiers, ce qui crée un conflit de fusion. Vous devez savoir comment gérer ces conflits. Nous allons donc en créer un délibérément et vous montrer comment le gérer. comment le gérer. REMARQUE: Le partage d'un dépôt de projet local (même par le biais d'un service de synchronisation de fichiers comme Dropbox) n'est pas recommandé, car les commits y sont généralement écrits directement, et vous avez donc très peu de protection contre les conflits. Pour générer notre conflit, nous devons d'abord avoir deux référentiels locaux, un pour le référentiel maître et un pour notre copie. Pour créer cela, nous devons cloner le référentiel, ce qui est fait avec les étapes suivantes : Allez dans Source Control > Clone Repository. Dans le champ URL, saisissez le chemin d'accès au projet que nous avons configuré dans le guide sur l'inversion des fichiers ci-dessus (par défaut, ce sera :\Users\\Documents\GameMakerStudio2\). Nous devons ensuite définir un chemin pour le clonage : Lorsque nous cliquons sur le bouton "Okay", nous obtenons l'avertissement suivant concernant la tentative de cloner un référentiel "non nu" : Nous pouvons cliquer sur "Okay" ici encore. On va maintenant nous présenter un explorateur de répertoire pour créer un dépôt proxy "nu" (c'est un dépôt que vous pouvez stocker en toute sécurité sur Dropbox, par exemple). Naviguez jusqu'à l'endroit où nous voulons créer ce dépôt de proxy et sélectionnez (ou créez) le dossier à utiliser. Une fois que nous avons fait cela, GameMaker Studio 2 va pousser le projet d'origine vers le proxy, et en tirer un nouveau vers le chemin que nous avons défini dans la fenêtre Clone Repository, puis ouvrir automatiquement un navigateur de fichiers pour que nous puissions ouvrir le projet stocké. stocké. Si nous allons maintenant dans Source Control > View History, nous verrons nos commits originaux plus un nouveau "Push For Clone". Vous avez maintenant cloné votre référentiel et vous pouvez continuer à travailler sur la version locale du projet.
