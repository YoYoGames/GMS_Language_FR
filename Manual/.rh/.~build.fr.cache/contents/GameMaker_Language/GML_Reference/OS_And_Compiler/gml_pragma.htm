gml_pragma La fonction gml_pragma affecte la façon dont la cible donnée compile votre code et doit être appelée avec les différentes commandes pour optimiser davantage la compilation finale de votre projet. Ces commandes sont effectivement pré-traitées avant que le et donc la fonction peut être placée n'importe où dans votre projet et elle sera toujours traitée avant que le jeu ne soit entièrement compilé. Les commandes disponibles sont les suivantes : " forceinline" - Lorsque la fonction est appelée avec le pragma " forceinline " dans une fonction de script, le YYC compilera le projet avec la fonction en ligne, plutôt que référencée. Cela donne un avantage supplémentaire au traitement mais il convient d'être prudent lors de son utilisation car cela augmentera également la taille du fichier exécutable final, en particulier si le code en ligne est important et/ou utilisé à plusieurs endroits différents, et augmentera considérablement le temps de compilation. Notez que ce pragma n'est valable que lors de la construction à l'aide de YYC. "global", "[code gml]" - Le pragme"global" vous permet d'appeler un code GML formaté sous forme de chaîne , à une portée globale, au moment de la compilation, avant l'exécution de la première pièce du jeu. Par exemple gml_pragma("global", "Init()") ; Cela appellera la fonction de script "Init" avant l'exécution de la première pièce du jeu. Notez que le GML fourni comme second argument doit être une constante de compilation, et notez également que vous ne pouvez pas utiliser ce pragme pour créer des instances ou effectuer toute opération nécessitant le fonctionnement d'une room (ou de tout ce qui se trouve dans une room). "PNGCrush" - Le pragme "PNGCrush" utilisera le programme PNGCrush sur chaque texture créée. Notez que cela peut augmenter de manière significative le temps de pour compiler le jeu, donc vous ne voulez pas l'utiliser tout le temps, bien que cela puisse faire des économies significatives sur la taille finale du fichier. "Texgroup.Scale", "[TextureGroupName]", " [Scale Divisor]" - Le "Texgroup.Scale" mettra à l'échelle le groupe de texture donné lors de la compilation. Vous devez fournir deux arguments supplémentaires ici : le "[TexGroupName]", qui est le nom (une chaîne de caractères) du groupe de texture à mettre à l'échelle, et le "[Scale Divisor]" (également une chaîne de caractères), qui est le diviseur que vous souhaitez utiliser pour la mise à l'échelle, c'est à dire gml_pragma("Texgroup.Scale", "level1", "2") ; Ainsi, toutes les textures du groupe de textures "level1" seront divisées par deux. "UnityBuild", "[enable/disable]" - Si vous appelez la fonction avec ce pragme et que vous définissez l'argument enable/disable à true, lors de la compilation , tous les fichiers .cpp du projet seront regroupés dans un seul fichier. qu'il utilise ensuite pour tout construire, par exemple : gml_pragma("UnityBuild", "true") ; L'avantage de faire un unity build est que les constructions sont plus rapides, mais l'inconvénient est qu'il fait une construction complète à chaque fois, donc même si vous changez une seule partie du code, il reconstruira tout sans utiliser les fichiers cachés. Ceci a été a été ajouté spécifiquement pour l'exportation Xbox One en utilisant le YYC bien qu'il puisse être appelé pour d'autres constructions(YYC seulement). Pour plus d'informations sur les builds de l'unité, veuillez voir ici. NOTE: Le premier argument de la fonction gml_pragma doit être une constante de chaîne de caractères au moment de la compilation et non une variable. Syntaxe : gml_pragma(commande, [optionnel...]) ; Arguments Description commande Une chaîne de caractères avec l'une des commandes énumérées ci-dessous. [facultatif] Certaines des commandes disponibles nécessitent un ou plusieurs arguments facultatifs. Ceux-ci sont expliqués ci-dessous pour chaque commande. Les retours : N/A Exemple : gml_pragma("forceinline") ; L'exemple de code ci-dessus forcera la fonction de script où elle est utilisée à être en ligne à la compilation.
